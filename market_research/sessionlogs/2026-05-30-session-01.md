### A.4 attempt #1 — FAILED (mistake M2 + M3)

**What I did**: Spawned 1 agent to do all 3 stocks × 4 dates in sequence (~25–40 min estimate). Foreground mode. No checkpoint protocol. No incremental file writes — agent was instructed to write the final output at the end.

**What happened**: Agent ran for ~55 minutes. Pranav interrupted because he had no visibility into what was happening (foreground agents don't stream progress to chat). The interrupt killed the agent. **All ~55 minutes of computation was lost** — no partial file written, no recovery possible.

**My compounding error (M3)**: I received a "tool use was rejected" message after the interrupt. I told Pranav: "the agent hasn't started — interrupted before it ran." This was wrong. Pranav (rightly) called it out: *"was it done? it was running for 55 mins, The agent hasn't started — really? then what was running for 55 mins?"*

**The correction**: I checked the filesystem (no `12_cycle_A_test_v0.1.md` existed), confirmed the agent had run but its work was lost, apologized for the misread, and reset task #19 to pending.

**Lessons learned** (now codified in `LEARNINGS.md`):
1. Any agent > ~5 min must use **incremental checkpoint writes** — write to disk after every logical step, not just at end.
2. Long agents should run in **background mode** (`run_in_background: true`) so they don't block the chat and Pranav can `Read` the file anytime.
3. When something looks ambiguous, **check the filesystem before making confident claims**. Don't state "X didn't happen" — verify first.

### The checkpoint protocol (post-mortem design)

After the failure, I designed and proposed a new agent pattern:

```
Agent uses Write tool to:
  1. Initialize the output file with a skeleton structure + status line:
     `🔄 Initialized — about to begin <task>`
  2. After every major computational step (each ratio, each date), use Write
     to overwrite the file with everything accumulated so far PLUS updated status:
     `🔄 <Current sub-task in progress>`
  3. At end, status = `✅ All <X> complete.`
  4. Always update the "Last updated" timestamp.

Pranav (or Claude) can `Read` the file at any time mid-run to see live progress.
If the agent dies, work up to the last checkpoint is preserved (atomic Write
operations mean we never get half-written files).
```

Combined with `run_in_background: true`, this gives:
- Non-blocking chat during long agent runs
- Live progress visible on disk
- Atomic crash safety (only the in-flight step's work is lost on interrupt)
